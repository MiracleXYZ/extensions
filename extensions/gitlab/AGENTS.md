## Learned User Preferences

- When changing this Raycast GitLab extension, preserve existing code style, structure, and behavior unless explicitly requested; when resolving merge conflicts or integrating upstream, prefer the user's architecture and solutions—do not refactor unrelated patterns or drift from local conventions.
- Run requested terminal, lint, and Raycast dev/debug commands yourself when asked instead of only describing them.
- Communicate in Russian in chat; code comments, UI copy, and docs follow existing project language.
- Never commit or push without explicit user request.
- Async mutations and errors: inline `try`/`catch` with `showToast({ style: Toast.Style.Animated, title })` before the action, `showToast(Toast.Style.Success, …)` after, and `showFailureToast` from `@raycast/utils` in `catch`—no shared toast wrapper helpers. Skip for Refresh/navigation-only actions. In `confirmAlert`, mark primary actions as destructive only for truly destructive delete/close/cancel flows, not ordinary state-changing confirmations like resolving a thread.
- Preferences: use typed `getPreferences()` from `utils.ts` (`Preferences` interface with all `package.json` keys); no per-key helpers or casts at call sites.
- API/GraphQL types and selections: include only fields used in UI and actions; do not expand response schemas with unused properties during refactors.
- Data fetching: prefer `useCachedPromise`, `useCachedState`, and `Cache` from `@raycast/utils` over custom cache hooks; prefer `usePromise` over `useState`+`useEffect` when async fetch is the only state update path; pass `initialData` to `useCachedPromise` where possible (e.g. `[]` for lists) to simplify UI and avoid nullable data guards; handle errors inline in UI (not toast-only `useEffect`)—`usePromise`/`useCachedPromise` show default error toasts unless errors are handled at the call site without suppressing them; do not add custom `onError` on these hooks solely to show toasts. Do not poll list views with `useInterval` or similar timers—refresh via explicit actions only. Menu Bar commands keep Raycast's built-in command `interval` in `package.json` (`mrmenu`, `issuemenu`, `todomenubar`).
- React style: declare JSX inline—do not store elements in `useMemo`; PascalCase for functions that return components; inline 1–3 line single-use expressions (except enum/mapping tables); omit redundant guards when types/API contracts already guarantee values. Do not extract one-off 1–3 line logic into named helpers (`flatListViewPreferences`, `getIcon`, `clipText`, `navTitle`, `stateColor`, …)—put it at the call site instead. Keep a helper only when reused, a mapping table, or the logic is long enough that a name genuinely helps. Memoize genuinely expensive derived data in `List`, `List.Item`, and `Form` components with `useMemo`—sorting, `Set`/`Map` merges/dedup, markdown/formatting, and non-trivial array/object assembly—so work is not repeated every render; do not memoize simple ternaries or one-step reads of already-stored state/values.
- Locals: do not introduce intermediate variables for obvious single-use values—inline them at the call site (e.g. pass `` `mr_pipelines_${mr.project_id}_${mr.iid}` `` directly as `cacheKey` instead of `const cacheKey = ...`). Prefer a single property chain (`foo.bar.baz`) over stepwise extraction across several locals (`const x = foo; const y = x.bar; ...`) when the chain is short and readable. Keep a `const` only when the value is reused, the expression is complex enough to warrant a name, or a local genuinely clarifies non-obvious logic—not to decompose a clear one-step access.
- Naming: use full words for parameters and variables—no single-letter names (`s`, `e`, `v`, `i`, …) and no opaque abbreviations (`val`, `pref`, `prefs`, `np`, `brch`, `rp`, …). Prefer domain terms (`mergeRequest`, `preference`, `value`, `username`) or descriptive names (`statusCode`, `nextPresets`); don't over-lengthen when local context already makes a shorter full word sufficient (`branch`, `project`).
- Dates/durations: `formatDate` for relative display, `formatDateTime` for absolute strings (tooltips, etc.), and `formatDurationHuman` for compact CI duration text in `utils.ts`; avoid ad-hoc `toLocaleString` or legacy date/duration helpers.
